# PiTerm 代码审查与优化建议报告

本报告针对 PiTerm 项目的架构设计、安全性、性能及代码质量进行了深度分析，识别出若干核心缺陷并提出了相应的优化方向。

---

## 1. 核心缺陷分析

### 🔴 安全性缺陷：私钥落盘风险
- **现状**：在 `src-tauri/src/commands/ssh/core.rs` 的 `establish_base_session` 函数中，私钥认证是通过将私钥内容写入临时文件（`ssh_key_{id}.pem`）实现的，认证后再删除。
- **风险**：这是严重的安全性隐患。如果程序异常崩溃或删除失败，敏感私钥将残留在磁盘。此外，临时目录通常对其它进程可见，存在被截获风险。
- **优化方向**：使用 `libssh2` 提供的内存认证接口（如 `userauth_pubkey_memory`），直接从内存读取私钥字符串进行认证，完全避免文件操作。

### 🔴 性能缺陷：读取线程的“忙等”轮询
- **现状**：`spawn_shell_reader_thread` 采用 `loop` 配合 `thread::sleep(Duration::from_millis(10))` 的方式循环尝试读取 SSH 数据。
- **问题**：这种方式每秒会唤醒 CPU 100 次，即便终端没有任何输出。这会导致不必要的能量消耗，尤其是在笔记本电脑上。
- **优化方向**：迁移到 **异步 I/O (Async I/O)** 架构。结合 Tauri 的 `tokio` 运行时和 `async-ssh2` 等库，利用操作系统的事件通知（epoll/kqueue）来驱动数据读取。若维持同步逻辑，也应改为带超时的阻塞读取。

### 🟡 稳定性缺陷：连接生命周期管理不完善
- **现状**：连接状态保存在全局 `HashMap` 中。虽然有 `disconnect_ssh` 命令，但依赖前端主动调用。
- **问题**：如果前端崩溃、网络闪断或发生竞态条件，后端可能残留大量的僵尸 TCP 连接和 SSH 会话，占用系统句柄。
- **优化方向**：
  1. 为 `SshConnection` 实现 `Drop` Trait，确保销毁时自动关闭 Channel、Session 和 TCP 句柄。
  2. 实现 **心跳机制 (Keep-alive)**：目前虽然设置了超时，但缺乏主动的心跳包发送逻辑。

### 🟡 协议实现缺陷：缺乏连接复用 (Multiplexing)
- **现状**：每个终端标签页会同时建立 3 条独立的 TCP 连接（Shell、Monitor、SFTP）。
- **问题**：资源开销大，且容易触碰服务器的连接频率限制（如 Fail2ban）。
- **优化方向**：实现 **单连接多路复用**。只需建立一个 `ssh2::Session`，然后在该 Session 上开启多个 `Channel` 运行不同业务。

### 🟡 安全逻辑缺陷：主机密钥确认 (Known Hosts) 存在竞态风险
- **现状**：`trust_host_key` 命令通过再次发起连接来获取主机指纹。
- **问题**：存在中间人攻击 (MitM) 风险。如果在“检查”到“信任”的间隙内攻击者切换了密钥，程序会信任错误的密钥。
- **优化方向**：在首次校验时将指纹缓存在内存中，待用户确认后直接将缓存值写入磁盘，严禁二次连接获取。

### 🔴 功能缺陷：代理配置 (Proxy) 未生效
- **现状**：数据库和 UI 中存在代理设置，但在 `connect_ssh` 核心流程中完全未被引用。
- **问题**：用户配置的代理设置无效，无法连接内网或跳板机后的服务器。
- **优化方向**：在 `establish_base_session` 阶段，通过 `proxies` 表检索对应设置，并在创建 `TcpStream` 前按需建立代理通道。

### 🟡 功能缺陷：不支持交互式身份验证 (2FA)
- **现状**：仅支持密码和私钥认证。
- **问题**：无法连接开启了双因子认证（2FA）或需要键盘交互输入的服务器。
- **优化方向**：实现 `userauth_keyboard_interactive` 处理逻辑。

### 🟡 体验缺陷：PTY 终端模拟类型硬编码
- **现状**：硬编码为 `request_pty("xterm", ...)`。
- **问题**：在某些复杂的 CLI 工具中会产生颜色显示或布局异常（如 256 色显示）。
- **优化方向**：允许用户自定义 `TERM` 环境变量（如 `xterm-256color`）。

## 2. 优化方向指南

### 🚀 响应式体验优化
- **终端 Resize 优化**：目前的 Resize 采用 `setTimeout` 延迟触发，会导致窗口缩放时终端内容闪烁。建议使用 `ResizeObserver` 配合精准的 `fit()` 调用，并在后端实现更平滑的 PTY Resize 响应。
- **数据传输批处理**：在高频率数据输出（如 `cat` 大文件）时，频繁跨进程通信（IPC）会拖累 UI 渲染。建议在后端对高频输出进行缓冲区聚合。

### 🔐 增强型安全性
- **Vault 加密算法升级**：目前 Vault 系统依赖用户 Master Key，应确保数据库中的敏感数据采用高强度的 AES-256-GCM 且带有动态 IV（初始化向量）。
- **进程权限收敛**：利用 Tauri 的 `capabilities` 进一步限制前端只能调用必要的系统资源。

### 🛠 代码质量提升
- **结构化错误处理**：将目前大量的 `Result<T, String>` 替换为基于 `thiserror` 的自定义错误枚举，以便前端能针对“认证失败”、“超时”、“域名无法解析”显示不同的友好提示。
- **前端状态机化**：`useTerminalSession` 钩子逻辑分散，建议引入简单的有限状态机（FSM），清晰定义 `CONNECTING`, `ESTABLISHED`, `RECONNECTING`, `TERMINATED` 等状态迁移逻辑。

### 🌍 国际化与扩展性
- **全量 Locale 审计**：代码中仍存在部分硬编码的中文提示，需统一收录到 `locales` 中。
- **插件系统预研**：考虑将“高亮规则”、“快捷指令”模块抽象化，为未来支持第三方插件做准备。

# PiTerm 代码审计与优化建议报告 (2026-04-18)

本报告基于对 `PiTerm` 核心代码库（Rust 后端与 React 前端）的深度分析，旨在识别现有的架构缺陷、安全风险及性能瓶颈，并提供针对性的优化建议。

---

## 1. 安全性审计 (Security Audit)

### 1.1 凭据存储风险 (Credential Storage)
- **现状分析**：在 `src-tauri/src/db.rs` 的 `servers` 表定义中，依然保留了 `password`、`private_key` 和 `passphrase` 字段。虽然引入了 Vault 金库系统（通过 `password_id` 和 `key_id` 关联），但目前的代码逻辑（如 `ssh/mod.rs` 第 427/501 行）允许在未启用金库时将明文密码/私钥存入数据库。
- **潜在风险**：如果用户未开启金库，敏感凭据将以明文形式存储在本地 SQLite 数据库中。一旦本地文件系统被攻击，所有服务器权限将直接暴露。
- **建议**：
  - 强制执行金库加密，移除 `servers` 表中的明文凭据字段。
  - 对于“快速连接”，应仅在内存中处理凭据，禁止落盘。

### 1.2 弱加密算法支持 (Legacy SSH Algorithms)
- **现状分析**：在 `ssh/core.rs` 的 `configure_legacy_algorithms` 函数中，默认启用了大量过时的交换算法（如 `diffie-hellman-group1-sha1`）和加密方法（如 `3des-cbc`）。
- **潜在风险**：虽然满足了旧设备的兼容性需求，但对现代服务器而言，这降低了整体安全水位。
- **建议**：
  - 提供全局或单服务器级别的“兼容模式”开关，而非默认对所有连接开启弱算法。

### 1.3 Shell 命令注入风险 (Shell Injection)
- **现状分析**：SFTP 实现中的 `chmod -R` 操作（`sftp_impl.rs` 第 184 行）使用了字符串格式化构建 Shell 命令：`format!("chmod -R {:03o} '{}'", mode_num, safe_path)`。
- **潜在风险**：虽然代码尝试转义了单引号，但依赖字符串过滤来防止注入在复杂的 Path 或特殊的远程 Shell 环境下可能失效。
- **建议**：
  - 尽量使用原生 SFTP 协议操作。对于必须使用 Shell 的递归操作，应极其慎重地验证路径字符集，或改为分步 SFTP 递归处理。

### 1.4 Master Key 内存安全
- **现状分析**：`vault.rs` 中的 `VaultState` 直接在内存中持有解密后的 `Key<Aes256Gcm>`。
- **建议**：
  - 使用 `secrecy` 库或类似工具包裹敏感密钥，确保在 Drop 时主动擦除内存，防止内存 Dump 攻击。

## 2. 性能与架构瓶颈 (Performance & Architecture)

### 2.1 严重缺陷：三重连接问题 (The Triple Connection Issue)
- **现象描述**：在 `src-tauri/src/commands/ssh/mod.rs` 的 `connect_ssh` 函数中，每次连接服务器都会分别调用 `create_shell_channel`、`create_monitor_session` 和 `create_sftp_session`。
- **架构缺陷**：这三个函数都会调用 `establish_base_session`，导致**对同一台服务器发起三次独立的 TCP 连接和三次重复的 SSH 握手**（Auth + Handshake）。
- **影响**：
  - 连接速度慢（延迟翻倍）。
  - 连接建立时瞬时资源消耗高。
  - 易触碰服务器端的并发连接限制（MaxStartups）或防火墙频率限制。
- **建议**：
  - 实行**会话复用（Multiplexing）**。在同一个 SSH 会话（Session）上开启多个通道（Channel），一个用于 Shell，一个用于 SFTP 子系统，一个用于监控命令。

### 2.2 同步 I/O 阻塞风险
- **现状分析**：由于使用的 `ssh2-rs` 底层基于 `libssh2`（同步库），所有的 I/O 操作（包括 SFTP 文件读写、终端读取线程等）都是阻塞式的。
- **影响**：虽然在 Rust 层使用了 `spawn_blocking`，但大量的线程创建和 Mutex 竞争可能导致在大文件传输或高频终端输出时 UI 响应变慢。
- **建议**：
  - 在大文件传输（Upload/Download）中引入任务队列和进度反馈。
  - 考虑未来迁移到 `russh` 等异步 SSH 库，以获得更好的并发性能。

### 2.3 数据库访问模式优化
- **现状分析**：SQL 语句散落在 `commands/*.rs` 各处。
- **建议**：
  - 将数据库操作下沉到 `models` 或 `dal`（数据访问层），统一错误处理和事务逻辑。

## 3. 代码质量与可维护性 (Maintainability)

### 3.1 错误处理的一致性
- **现状**：大量使用 `.map_err(|e| e.to_string())`。
- **影响**：前端接收到的错误只是普通的字符串，难以根据错误类型（如：认证失败 vs 网络超时 vs 金库锁定）进行差异化 UI 处理。
- **建议**：
  - 定义结构化的错误枚举（Enum），并实现 `serde::Serialize`，让前端能够精确识别错误码。

### 3.2 终端数据流压力
- **现状**：终端读取线程通过 `app.emit` 将数据推送到前端。
- **优化点**：
  - 缺乏背压控制。当输出海量文本（如 `cat` 大文件）时，前端渲染可能卡死。
  - 应考虑在后端增加流量整流，或在前端引入 Buffer 机制。
